# Retrieval-Augmented Generation (RAG) Notes: A Practical Tutorial

This tutorial explains how retrieval-augmented generation works, why it is useful,
how it is implemented in this repository, and how to evaluate and improve a RAG
system.

## 1. What problem does RAG solve?

A language model has two important limitations:

1. Its learned knowledge may be incomplete, stale, or unsuitable for a private
   collection of documents.
2. Its context window is limited. Sending every document with every question is
   expensive and makes relevant evidence harder to identify.

RAG addresses these limitations by retrieving a small set of relevant passages at
question time and giving those passages to the model as context.

The basic flow is:

```text
Documents
   ↓
Load and clean
   ↓
Split into chunks
   ↓
Create vector representations and store them
   ↓
User question
   ↓
Retrieve the closest chunks
   ↓
Add retrieved context to the model input
   ↓
Generate an answer grounded in the retrieved evidence
```

RAG is therefore not a replacement for a language model. It is a retrieval and
context-construction layer around one.

## 2. RAG terminology

### Corpus

The collection of source material available to the system: PDFs, web pages,
course notes, policies, tickets, database rows, or other records.

### Document

A logical source item, such as one PDF or one text file. A document normally has
metadata such as its title, author, date, URL, or access permissions.

### Chunk

A smaller passage extracted from a document. Retrieval usually operates on chunks
because a whole document is often too large and too imprecise to use as model
context.

### Embedding

An embedding is a vector representation of text. Texts with related meaning tend
to be close together in vector space. The embedding model is used both when
indexing chunks and when encoding a user query.

### Vector store

A database or index that stores chunk vectors, source text, and metadata. This
repository uses ChromaDB as a local persistent vector store.

### Retrieval

The process of finding chunks that are likely to help answer a question. A basic
system uses nearest-neighbor vector search. More advanced systems combine vector
search, keyword search, metadata filters, and reranking.

### Grounding

Using retrieved evidence to constrain the answer. A well-grounded answer should
distinguish what the retrieved sources support from what they do not support.

## 3. Two meanings of “the model retrieves”

It is important to distinguish the retrieval mechanism from the agent’s decision
to use it.

### Application-managed RAG

The application retrieves before calling the model:

```text
User question
  → application calls retrieve(...)
  → application inserts chunks into the model input
  → model answers
```

This is still RAG. Retrieval is deterministic from the application’s point of
view, and the model receives augmented context.

That is the design in [`01_rag_in_context`](01_rag_in_context/). In
[`agent.py`](01_rag_in_context/agent.py), `CourseAssistant.ask()` calls
`retrieve()` and builds a prompt containing `Course-note context` before calling
the model.

### Tool-based or agentic RAG

The model receives a retrieval tool and decides whether to call it:

```text
User question
  → model decides whether search_course_notes is useful
  → model calls the retrieval tool when appropriate
  → tool returns relevant chunks
  → model uses the tool result to answer
```

That is the design in [`02_rag_tool`](02_rag_tool/). The application registers
`search_course_notes`; the model may call it through the agent tool loop.

The two designs answer different teaching questions:

| Design | Retrieval decision | Main advantage | Main risk |
|---|---|---|---|
| Application-managed | Application code | Predictable and simple | Retrieves even when unnecessary |
| Tool-based | Language model | Flexible and selective | The model may fail to call the tool or call it poorly |

Both are RAG systems. The distinction is who controls retrieval.

## 4. The indexing pipeline

Indexing is the offline or startup process that makes retrieval possible.

### Step 1: Load source documents

The Part 1 helper loads non-empty `.txt` files from its `documents/` directory:

```python
def load_documents(folder: str | Path) -> list[tuple[str, str]]:
    documents = []
    for path in sorted(Path(folder).glob("*.txt")):
        text = path.read_text(encoding="utf-8").strip()
        if text:
            documents.append((path.stem, text))
    return documents
```

In a production system, this stage also needs to handle parsing, encoding errors,
duplicate files, access permissions, and source metadata.

### Step 2: Split documents into chunks

The repository’s simple chunker preserves paragraph boundaries and aims for a
target size of 700 characters:

```python
def chunk_text(text: str, size: int = 700) -> list[str]:
    chunks = []
    current = ""
    for paragraph in text.split("\n\n"):
        candidate = f"{current}\n\n{paragraph}".strip()
        if current and len(candidate) > size:
            chunks.append(current)
            current = paragraph
        else:
            current = candidate
    if current:
        chunks.append(current)
    return chunks
```

Chunking is one of the most important quality decisions in RAG.

If chunks are too large, retrieval returns too much unrelated text and wastes
context. If chunks are too small, important definitions and their explanations may
be separated. Useful chunking strategies include:

- paragraph or section boundaries for well-structured notes;
- token-based windows for consistently sized chunks;
- sliding windows with overlap when information crosses boundaries;
- structure-aware splitting for Markdown, HTML, code, or legal documents;
- parent-child retrieval, where a small chunk is searched but a larger section is
  returned to the model.

There is no universally correct chunk size. Measure retrieval quality on your own
questions.

### Step 3: Assign stable IDs and metadata

The repository uses IDs such as `source-0` and stores metadata containing the
source name and chunk number. Production metadata often includes:

- source document ID and title;
- section heading and page number;
- publication or update date;
- tenant, department, or access-control scope;
- document version;
- URL or file path.

Stable IDs make re-indexing safer and allow a changed document to replace its old
chunks rather than creating duplicates.

### Step 4: Create and store embeddings

Each chunk is converted into a vector and stored with its text and metadata. At
query time, the question is embedded using the compatible query path and compared
with the stored vectors.

The indexing and query embedding configuration must be compatible. Changing the
embedding model generally requires re-indexing the corpus.

### Step 5: Persist the index

Part 1 opens a persistent Chroma collection and refreshes its source documents in
`open_collection()`. The generated `.chroma/` directory is local state and is
ignored by Git in this project.

## 5. The query pipeline

At question time, a RAG application normally performs these steps:

1. Receive and normalize the question.
2. Apply access-control and metadata filters.
3. Search for a larger candidate set than will ultimately be shown.
4. Optionally rerank candidates with a stronger relevance model.
5. Select a final context budget.
6. Format the passages with source labels.
7. Ask the model to answer using that evidence.

The repository’s basic retrieval function is:

```python
def retrieve(collection: object, question: str, limit: int = 3):
    result = collection.query(
        query_texts=[question],
        n_results=limit,
        include=["documents", "metadatas", "distances"],
    )
    return [
        {"text": text, "source": metadata["source"], "distance": distance}
        for text, metadata, distance in zip(
            result["documents"][0],
            result["metadatas"][0],
            result["distances"][0],
            strict=True,
        )
    ]
```

The `distance` value is useful for diagnostics, but its exact interpretation
depends on the vector-store distance configuration. Do not automatically describe
it as a probability or confidence score.

## 6. Constructing model context

Retrieved passages should be clearly separated and labelled:

```text
Course-note context:
[rag_course]
Retrieved passage one...

[agent_course]
Retrieved passage two...

User question: How does retrieval work?
```

Useful grounding instructions tell the model to:

- use the supplied sources when they answer the question;
- distinguish source-supported facts from general knowledge;
- say when the sources do not answer the question;
- avoid inventing citations or unsupported details;
- cite the source name, page, or section when available.

Retrieved context is evidence, not automatically truth. Documents can be outdated,
incorrect, malicious, or irrelevant. The model still needs instructions and the
application needs validation.

## 7. Conversation history and RAG context are different

Conversation history answers: “What did the user and assistant say earlier?”

Retrieved context answers: “What external evidence is relevant to this question?”

They should be considered separately. A common design is:

```text
System instructions
Conversation history
Current retrieved context
Current user question
```

If retrieved passages are stored inside every historical user message, later turns
may resend old evidence even when it is no longer relevant. This increases token
usage and can confuse the model. Alternatives include retrieving fresh context for
each turn, storing only the user and assistant messages, or explicitly marking
retrieved context as turn-scoped.

The trace logs in this repository show the expanded input sent to the model. They
are useful for seeing exactly where application-managed RAG inserts evidence.

## 8. Retrieval quality techniques

### Query rewriting

A conversational question such as “What about the second one?” may be impossible
to retrieve accurately without the prior context. Rewrite it into a standalone
query before searching:

```text
Conversation: The user asked about vector indexes.
Follow-up: What about the second one?
Search query: What are the limitations of the second vector-index approach?
```

Keep the rewritten query separate from the user’s original wording so the original
question remains available to the final answer step.

### Hybrid search

Vector search is good at semantic similarity. Keyword search is often better for
exact names, product codes, error messages, and legal terms. Hybrid retrieval
combines both signals.

### Metadata filtering

Filter before or during retrieval when the question has a known scope, such as
department, date range, document type, or user permissions. Access control must be
enforced by the application or retrieval layer, not left to the model.

### Reranking

Retrieve perhaps 10–50 candidates, then use a reranker or a stronger relevance
model to select the best few. This often improves precision without putting every
candidate into the model context.

### Diversity

The top results may all be near-duplicates from one document. Diversity methods
can select passages that cover different relevant sources or sections.

### Context compression

Long passages can be compressed or filtered, but compression must not remove the
qualifications needed to interpret a claim. Preserve source identity and enough
surrounding context to avoid misleading excerpts.

## 9. Common RAG failure modes

### Retrieval misses the answer

Possible causes include poor chunking, vocabulary mismatch, a vague query, bad
embeddings, or an overly small result limit.

Investigate by logging the query, retrieved source IDs, distances, and passages.
Try query rewriting, hybrid search, better chunking, or a larger candidate set.

### Retrieval returns plausible but irrelevant text

Semantic similarity is not the same as answer relevance. Add reranking, metadata
filters, source weighting, or a minimum relevance threshold.

### The answer ignores the retrieved context

The prompt may be ambiguous, the context may be poorly formatted, or the model may
have too much competing conversation history. Use clear source delimiters and ask
the model to state when evidence is insufficient.

### The model invents an answer

Retrieval does not automatically prevent hallucinations. Require evidence-based
answers, expose source labels, and evaluate unsupported-answer behavior explicitly.

### The model cites a source incorrectly

Carry source metadata alongside every chunk. Do not ask the model to reconstruct
page numbers or URLs that were not provided in context.

### Prompt injection in retrieved documents

Retrieved text is untrusted data. A document can contain instructions such as “ignore
the system message.” Tell the model to treat retrieved content as evidence, not as
instructions, and consider content scanning, source trust levels, and tool/action
approval boundaries.

### Sensitive information is retrieved

Apply access checks before retrieval and avoid relying on prompt instructions to
protect confidential data. Log safely: redact credentials and review whether user
questions or retrieved documents may themselves contain sensitive information.

## 10. Evaluating a RAG system

Evaluation should measure retrieval and generation separately.

### Retrieval metrics

Create questions with known supporting chunks and measure:

- Recall@k: was a supporting chunk retrieved in the top `k`?
- Precision@k: how many top results were relevant?
- MRR: how early did the first relevant result appear?
- nDCG: how well were results ordered by relevance?

### Generation metrics

Evaluate whether the answer is:

- correct relative to the source;
- supported by the retrieved evidence;
- complete enough for the question;
- clear about uncertainty;
- free of unsupported claims;
- properly cited when citations are required.

### A practical evaluation set

Build a small, versioned dataset containing:

```text
question
expected source IDs
key facts that should appear
facts that must not be invented
acceptable answer notes
```

Include easy questions, ambiguous questions, follow-ups, questions with no answer
in the corpus, exact-keyword questions, and adversarial or injection-like documents.

## 11. Cost, latency, and context budgeting

RAG introduces work before generation:

- query embedding;
- vector or hybrid search;
- optional reranking;
- context formatting;
- additional input tokens.

Measure end-to-end latency and token usage. More retrieved chunks are not always
better. Select a context budget that leaves enough room for the model’s answer and
keeps irrelevant material out.

Cache stable embeddings and repeated queries where appropriate. Re-index when the
corpus or embedding configuration changes, and retain document versions when
reproducibility matters.

## 12. When RAG is not the right solution

RAG is useful when the answer depends on a changing or private corpus. It may not
be the best fit when:

- the needed data is structured and should be queried with SQL;
- the task requires exact calculations better handled by code;
- the corpus is small enough to provide directly;
- the problem is behavior or style rather than factual knowledge;
- a deterministic rules engine can answer more reliably.

Many systems combine approaches: SQL for structured facts, APIs for current state,
RAG for explanatory documents, and a language model to coordinate and present the
result.

## 13. Learning path in this repository

Run the lessons in order:

1. **Part 1 — RAG in context**: the application retrieves on every turn and injects
   passages directly into the model input.
2. **Part 2 — Model-selected RAG tool**: the model chooses whether to call
   `search_course_notes`.
3. **Part 3 — MCP weather**: combines local RAG with a separate current-data tool.
4. **Part 4 — Summary subagent**: demonstrates multi-agent orchestration around
   retrieval and other tools.
5. **Part 5 — Markdown skill tool**: demonstrates loading reusable instructions as a
   tool result.

For each lesson, run:

```bash
cd 01_rag_in_context  # choose the lesson you want
python main.py --trace
```

The terminal shows the conversation. The generated log records the observable
model inputs, retrieval/tool exchanges, and responses. In particular, compare the
Part 1 and Part 2 logs to see the difference between application-managed retrieval
and model-selected tool retrieval.

## 14. A production-readiness checklist

- [ ] Sources are parsed, versioned, and assigned stable IDs.
- [ ] Chunking preserves enough semantic context.
- [ ] Embedding and query configurations are compatible.
- [ ] Access control is enforced before content reaches the model.
- [ ] Retrieval results retain source metadata.
- [ ] Hybrid search, filtering, or reranking is used where needed.
- [ ] The model is instructed to distinguish evidence from unsupported claims.
- [ ] No-answer and adversarial cases are evaluated.
- [ ] Retrieval and answer quality are measured separately.
- [ ] Token cost and latency are monitored.
- [ ] Logs are redacted and handled as potentially sensitive data.
- [ ] Re-indexing and rollback procedures are documented.
